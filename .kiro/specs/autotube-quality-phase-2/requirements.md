# Requirements Document

## Introduction

AutoTube Quality Phase 2 is a phased improvement plan for the AutoTube AI video generator. A recent pipeline test scored 28/50 (56%), and this feature addresses six improvement areas: render speed, YouTube metadata quality, thumbnail wiring, procedural background quality, render progress accuracy, and batch processing connectivity. All improvements must work within the existing codebase — no new external APIs may be added.

## Glossary

- **Renderer**: The `renderVideoToBlob` function in `src/services/videoRenderer.ts` that captures PNG frames and assembles them via ffmpeg or MediaRecorder.
- **ProceduralBackground**: The `drawProceduralBackground` function that draws animated gradients, particles, waves, and grid overlays onto the canvas each frame.
- **FrameSampleRate**: The number of frames captured per second of video (e.g. 4 for draft, 6 for standard).
- **OffscreenCanvas**: A secondary canvas used for compositing images before copying to the capture canvas.
- **VideoProject**: The central data object (`src/types.ts`) holding script segments, media assets, narration clips, and export settings for a single video.
- **ScriptSegment**: A single timed section of the video script with a type (`intro`, `section`, `transition`, `outro`), title, narration text, and duration.
- **MediaAsset**: A sourced image or video clip associated with a `ScriptSegment`.
- **NarrationClip**: A generated audio clip (OpenAI TTS or browser TTS) associated with a `ScriptSegment`.
- **YouTubeMetadata**: The title, description, tags, and chapter markers generated by `generateYouTubeMetadata` in `src/services/youtube.ts`.
- **SplitScreenThumbnail**: The 1280×720 thumbnail produced by `generateSplitScreenThumbnail` in `src/services/thumbnail.ts` that places a chart asset on the left and a portrait asset on the right.
- **BatchProcessor**: The `BatchProcessor` class in `src/services/batchProcessor.ts` and the `BatchProcessor` React component in `src/components/BatchProcessor.tsx`.
- **BatchJob**: A single entry in a batch queue with a topic, template config, and lifecycle status (`pending` → `running` → `complete` / `error`).
- **AssemblyStep**: The React component in `src/components/AssemblyStep.tsx` that displays render progress.
- **PreviewStep**: The React component in `src/components/PreviewStep.tsx` that shows the finished video and download/upload actions.
- **Store**: The `useVideoProject` hook in `src/store.ts` that manages all pipeline state.
- **AccentColor**: The segment-type-specific highlight colour used in the `draw` function (e.g. `#e74c3c` for intro, `#3498db` for section).
- **DataPoint**: A numeric value extracted from media asset metadata by `extractDataPoints` in `src/services/seoTitles.ts` (e.g. `$1.2T`, `+200%`, `2024`).

## Requirements

### Requirement 1: Render Speed — Reduce Particle Count During Rendering

**User Story:** As a content creator, I want the video to render faster, so that I can iterate on my videos without waiting several minutes per render.

#### Acceptance Criteria

1. WHEN `renderVideoToBlob` is called, THE Renderer SHALL pass an `isRendering` flag (or equivalent parameter) to `drawProceduralBackground` to indicate that a render is in progress.
2. WHILE `isRendering` is `true`, THE ProceduralBackground SHALL draw 30 particles instead of 120.
3. WHILE `isRendering` is `false` (preview/simulated mode), THE ProceduralBackground SHALL continue to draw 120 particles.
4. THE ProceduralBackground SHALL produce visually equivalent output at 30 particles as at 120 particles for the purposes of the final encoded video (particle size and alpha values may be adjusted to compensate for reduced count).
5. IF the `isRendering` parameter is absent or `undefined`, THEN THE ProceduralBackground SHALL default to the full 120-particle count.

### Requirement 2: Render Speed — Skip Film Grain During Rendering

**User Story:** As a content creator, I want the render to skip expensive per-frame effects that are imperceptible in the final video, so that render time is reduced without visible quality loss.

#### Acceptance Criteria

1. WHEN `renderVideoToBlob` is called, THE Renderer SHALL pass the `isRendering` flag to the `draw` function.
2. WHILE `isRendering` is `true`, THE `draw` function SHALL skip the film grain overlay loop entirely (the nested `for` loop over `grainStep` increments).
3. WHILE `isRendering` is `false`, THE `draw` function SHALL render the film grain overlay as before.
4. THE Renderer SHALL not skip any other visual element (letterbox bars, vignette, lower-third title, captions, progress bar) when `isRendering` is `true`.

### Requirement 3: Render Speed — Increase Frame Yield Interval

**User Story:** As a content creator, I want the renderer to yield to the browser event loop less frequently, so that render time is reduced without affecting output quality.

#### Acceptance Criteria

1. WHEN `renderVideoToBlob` is executing the frame capture loop, THE Renderer SHALL yield to the event loop (via `setTimeout(0)`) every 60 frames instead of every 30 frames.
2. THE Renderer SHALL still yield at least once per segment regardless of segment frame count.
3. IF the `signal` parameter is set and aborted, THEN THE Renderer SHALL check the abort signal at each yield point and throw a cancellation error.

### Requirement 4: Render Speed — Cache Procedural Background Per Segment

**User Story:** As a content creator, I want the renderer to avoid redrawing the static background elements every frame, so that render time is significantly reduced.

#### Acceptance Criteria

1. WHEN rendering begins for a segment, THE Renderer SHALL pre-render the procedural background at `progress = 0` to a dedicated offscreen canvas (the "background cache canvas") once per segment.
2. WHEN rendering subsequent frames within the same segment, THE Renderer SHALL blit the cached background canvas onto the offscreen canvas instead of calling `drawProceduralBackground` for every frame.
3. WHEN a new segment begins, THE Renderer SHALL invalidate the background cache and re-render it for the new segment's type and palette.
4. THE Renderer SHALL still apply all foreground elements (image, letterbox bars, vignette, grain, title, captions, progress bar) on top of the cached background each frame.
5. IF the background cache canvas cannot be created (e.g. `getContext('2d')` returns `null`), THEN THE Renderer SHALL fall back to calling `drawProceduralBackground` every frame without error.

### Requirement 5: Render Speed — Reduce Draft Frame Sample Rate

**User Story:** As a content creator, I want draft renders to complete faster, so that I can quickly preview the video structure before committing to a full-quality render.

#### Acceptance Criteria

1. WHEN `renderVideoToBlob` is called with `quality = 'draft'`, THE Renderer SHALL use a `frameSampleRate` of 3 frames per second.
2. WHEN `renderVideoToBlob` is called with `quality = 'standard'` or `quality = 'high'`, THE Renderer SHALL use the existing `frameSampleRate` values (6 and 8 respectively) unchanged.
3. THE Renderer SHALL pass the reduced `frameSampleRate` to the ffmpeg endpoint and MediaRecorder fallback so the output video plays at the correct speed.

### Requirement 6: YouTube Readiness — Rich Description Template

**User Story:** As a content creator, I want the auto-generated YouTube description to be structured and informative, so that my videos rank better in search and viewers know what to expect.

#### Acceptance Criteria

1. WHEN `generateYouTubeMetadata` is called, THE YouTubeMetadata generator SHALL produce a description that begins with a hook paragraph derived from the first `ScriptSegment`'s narration text (up to 300 characters).
2. THE YouTubeMetadata generator SHALL include a "What you'll learn" section containing one bullet point per `ScriptSegment` title (using the `•` character as the bullet).
3. THE YouTubeMetadata generator SHALL include a "Chapters" section using the output of `generateDetailedChapters(script)`.
4. THE YouTubeMetadata generator SHALL include a hashtag line at the end of the description containing at least 3 hashtags derived from the topic words and style.
5. THE YouTubeMetadata generator SHALL use only data already present in the `VideoProject` object (topic, style, script segments) — no external API calls.
6. THE YouTubeMetadata generator SHALL produce a description no longer than 5000 characters (YouTube's limit), truncating at a sentence boundary where possible.

### Requirement 7: YouTube Readiness — Smart Tag Generation

**User Story:** As a content creator, I want the auto-generated YouTube tags to include long-tail keyword variants, so that my videos are discoverable for a wider range of search queries.

#### Acceptance Criteria

1. WHEN `generateYouTubeMetadata` is called, THE YouTubeMetadata generator SHALL extract individual words longer than 3 characters from the topic string as base tags.
2. THE YouTubeMetadata generator SHALL generate long-tail tag variants for each base word in the format `"{word} explained"`, `"{word} documentary {year}"`, and `"{word} {style}"` where `{year}` is the current calendar year and `{style}` is the video style string.
3. THE YouTubeMetadata generator SHALL include the full topic string as a tag.
4. THE YouTubeMetadata generator SHALL include the video style string as a tag.
5. THE YouTubeMetadata generator SHALL deduplicate all tags (case-insensitive comparison) before returning.
6. THE YouTubeMetadata generator SHALL return no more than 15 unique tags.
7. IF the topic contains fewer than 2 words longer than 3 characters, THEN THE YouTubeMetadata generator SHALL supplement the tag list with generic tags (`"AI generated"`, `"documentary"`, `"explained"`) to reach at least 5 tags.

### Requirement 8: YouTube Readiness — Embed Data Points in Description

**User Story:** As a content creator, I want key numeric data points from my media assets to appear in the video description, so that viewers see compelling statistics before clicking.

#### Acceptance Criteria

1. WHEN `generateYouTubeMetadata` is called with a `VideoProject` that has media assets, THE YouTubeMetadata generator SHALL call `extractDataPoints(project.media)` to retrieve numeric data points.
2. IF `extractDataPoints` returns one or more data points, THEN THE YouTubeMetadata generator SHALL include a "Key Numbers" section in the description listing each data point on its own line.
3. IF `extractDataPoints` returns an empty array, THEN THE YouTubeMetadata generator SHALL omit the "Key Numbers" section entirely.
4. THE YouTubeMetadata generator SHALL place the "Key Numbers" section after the "What you'll learn" bullets and before the "Chapters" section.
5. THE YouTubeMetadata generator SHALL accept a `VideoProject` object (or at minimum `project.media` and `project.script`) as input in addition to the existing `title`, `topic`, and `script` parameters.

### Requirement 9: Thumbnail Wiring — Use Split-Screen Thumbnail in PreviewStep

**User Story:** As a content creator, I want the "Download Thumbnail" button to produce the best available thumbnail, so that I get a professional split-screen image instead of a plain single-image fallback.

#### Acceptance Criteria

1. WHEN the user clicks "Download Thumbnail" in `PreviewStep`, THE PreviewStep SHALL call `generateSplitScreenThumbnail(project, project.title)` instead of `generateThumbnail(project.title, project.topic, firstMedia?.url)`.
2. IF `generateSplitScreenThumbnail` throws an error, THEN THE PreviewStep SHALL fall back to calling `generateThumbnail(project.title, project.topic)` and use that result.
3. THE PreviewStep SHALL always produce a valid `Blob` from the thumbnail download action — it SHALL NOT leave the download handler in an unresolved or error state visible to the user.
4. THE PreviewStep SHALL pass the resulting `Blob` to `downloadThumbnail` with the filename `{sanitized_title}_thumbnail.png`.

### Requirement 10: Thumbnail Wiring — Show Thumbnail Preview Card

**User Story:** As a content creator, I want to see a preview of the generated thumbnail in the sidebar, so that I can evaluate it before downloading or uploading to YouTube.

#### Acceptance Criteria

1. WHEN the assembly step completes and the user navigates to `PreviewStep`, THE PreviewStep SHALL display a thumbnail preview card in the sidebar (the right-hand column of the 3-column grid).
2. THE thumbnail preview card SHALL show a 160×90 pixel image rendered from `generateSplitScreenThumbnail(project, project.title)` (or the fallback thumbnail if split-screen fails).
3. THE thumbnail preview card SHALL be displayed above the "Download Thumbnail" button.
4. IF thumbnail generation fails entirely, THEN THE PreviewStep SHALL display a placeholder card with the text "Thumbnail preview unavailable" instead of an error state.
5. THE thumbnail preview card SHALL be generated once when `PreviewStep` mounts and SHALL NOT regenerate on every render cycle.

### Requirement 11: Thumbnail Wiring — Pass Thumbnail to YouTube Upload

**User Story:** As a content creator, I want the split-screen thumbnail to be automatically included when I initiate a YouTube upload, so that I don't have to manually attach it in YouTube Studio.

#### Acceptance Criteria

1. WHEN the user clicks "Upload to YouTube" in `PreviewStep`, THE PreviewStep SHALL generate the split-screen thumbnail blob (using `generateSplitScreenThumbnail` with the same fallback chain as Requirement 9) before calling `openYouTubeUpload`.
2. THE PreviewStep SHALL pass the thumbnail blob as a `File` object (with `name` set to `{sanitized_title}_thumbnail.png` and `type` set to `image/png`) in the `thumbnail` field of the `YouTubeUploadConfig`.
3. IF thumbnail generation fails, THEN THE PreviewStep SHALL call `openYouTubeUpload` without a `thumbnail` field rather than blocking the upload flow.

### Requirement 12: Procedural Background Quality — Segment Title Card

**User Story:** As a viewer, I want to see the segment title prominently displayed when no image is available, so that I always know what topic is being discussed.

#### Acceptance Criteria

1. WHEN `draw` is called for a segment where no CORS-safe image is available (i.e. `asset` is `undefined` or `img.safeForCanvas` is `false`), THE Renderer SHALL render the segment title in the centre of the canvas in bold 72px white text.
2. THE Renderer SHALL draw a 4px horizontal underline below the title text using the segment's `AccentColor`, spanning 60% of the canvas width and centred horizontally.
3. THE Renderer SHALL render the title card in the vertical centre of the canvas (between the letterbox bars), not in the lower-third position.
4. THE Renderer SHALL still render the lower-third title overlay in its existing position regardless of whether the title card is shown.
5. IF the title text exceeds the canvas width minus 120px of horizontal padding, THEN THE Renderer SHALL wrap the text to a second line using the existing `wrapText` helper.

### Requirement 13: Procedural Background Quality — Data Stat Card

**User Story:** As a viewer, I want to see key statistics rendered as large on-screen cards when no image is available, so that the video communicates data visually even without media assets.

#### Acceptance Criteria

1. WHEN `draw` is called for a segment where no CORS-safe image is available AND the segment's `narration` text contains a number matching the regex `\d+`, THE Renderer SHALL extract the first matching number from the narration.
2. THE Renderer SHALL render the extracted number as a "stat card" in 96px bold white text, centred horizontally at 35% of the canvas height.
3. THE Renderer SHALL render a label below the stat number using up to 5 words surrounding the number in the narration text, in 24px regular white text.
4. IF both the title card (Requirement 12) and the stat card apply to the same frame, THEN THE Renderer SHALL render the stat card above the title card, with the stat card at 30% canvas height and the title card at 60% canvas height.
5. IF no number is found in the narration, THEN THE Renderer SHALL not render a stat card.

### Requirement 14: Procedural Background Quality — Accent-Coloured Letterbox Bars

**User Story:** As a viewer, I want the letterbox bars to use the segment's accent colour, so that the visual style feels more cohesive and dynamic.

#### Acceptance Criteria

1. WHEN `draw` renders the top and bottom letterbox bars, THE Renderer SHALL use the segment's `AccentColor` at 85% opacity (`rgba` with alpha `0.85`) instead of the fixed `rgba(0, 0, 0, 0.85)`.
2. THE Renderer SHALL derive the `AccentColor` from the same `accentColors` lookup already used for the lower-third underline and caption accent.
3. THE Renderer SHALL apply the accent-coloured bars to all segment types (`intro`, `section`, `transition`, `outro`).
4. IF the segment type is not found in the `accentColors` map, THEN THE Renderer SHALL fall back to `rgba(0, 0, 0, 0.85)`.

### Requirement 15: Render Progress UI — Per-Segment Progress Events

**User Story:** As a content creator, I want the render progress message to tell me which segment is currently being rendered, so that I can estimate how much time remains.

#### Acceptance Criteria

1. WHEN `renderVideoToBlob` begins rendering a segment, THE Renderer SHALL emit a progress event via `onProgress` with a message in the format `"Rendering segment {i+1}/{total}: {seg.title}"`.
2. THE Renderer SHALL emit this message at the start of each segment's frame loop (i.e. when `f === 0`).
3. THE Renderer SHALL continue to emit intermediate progress events within a segment (as it currently does every `totalFrames / 10` frames) using the same message format.
4. THE `AssemblyStep` component SHALL parse the segment index from the progress message using the pattern `Rendering segment (\d+)/(\d+)` to extract the current and total segment counts.
5. THE `AssemblyStep` component SHALL display the parsed segment progress (e.g. "Segment 3 of 10") below the main progress percentage.

### Requirement 16: Render Progress UI — Accurate Track Bar Mapping

**User Story:** As a content creator, I want the four render track bars to reflect real pipeline phases, so that I can see which part of the render is currently active.

#### Acceptance Criteria

1. WHEN `AssemblyStep` displays the render progress, THE AssemblyStep SHALL map the "Video Track" bar to frame capture progress, filling from 0% to 80% as `processingProgress` goes from 0 to 80.
2. THE AssemblyStep SHALL map the "Audio Track" bar to narration readiness: THE AssemblyStep SHALL fill the bar proportionally based on the count of `NarrationClip` entries with `status === 'ready'` divided by the total clip count.
3. THE AssemblyStep SHALL map the "Text Overlay" bar to mirror the "Video Track" bar (same value), representing caption rendering progress.
4. THE AssemblyStep SHALL map the "Effects" bar to ffmpeg assembly progress, filling from 80% to 100% as `processingProgress` goes from 80 to 100 (i.e. the bar is 0% until `processingProgress >= 80`, then fills linearly to 100%).
5. THE AssemblyStep SHALL receive the `project` prop (already available) to compute the "Audio Track" value from `project.narration`.

### Requirement 17: Render Progress UI — Per-Segment ETA

**User Story:** As a content creator, I want to see an estimated time remaining during rendering, so that I can decide whether to wait or cancel.

#### Acceptance Criteria

1. WHEN `AssemblyStep` is in the `processing` state and at least one segment has completed rendering, THE AssemblyStep SHALL display an estimated time remaining (ETA) below the progress bar.
2. THE AssemblyStep SHALL compute the ETA as: `(elapsed_seconds / segments_completed) * segments_remaining`, rounded to the nearest second.
3. THE AssemblyStep SHALL display the ETA in the format `"~{N}s remaining"` for values under 60 seconds, or `"~{M}m {S}s remaining"` for values of 60 seconds or more.
4. THE AssemblyStep SHALL start the elapsed time clock when `status` transitions to `processing`.
5. IF `segments_completed` is 0, THEN THE AssemblyStep SHALL display "Calculating..." instead of an ETA.

### Requirement 18: Batch Processing — Wire `onGenerate` in App.tsx

**User Story:** As a content creator, I want the "Generate All" button in the Batch Processor to actually run the full video pipeline for each job, so that I can produce multiple videos without manual intervention.

#### Acceptance Criteria

1. WHEN the `BatchProcessor` component calls `onGenerate` with a list of job configs, THE App SHALL invoke a `batchGenerate` function that runs the full pipeline (`generateScript` → `sourceMedia` → `generateNarration` → `assembleVideo`) for each job sequentially.
2. THE `batchGenerate` function SHALL process jobs one at a time (concurrency = 1) to avoid memory exhaustion from simultaneous canvas renders.
3. THE `batchGenerate` function SHALL be implemented in the Store (`useVideoProject`) and exposed as a named function.
4. THE App SHALL pass `batchGenerate` as the `onGenerate` prop to `BatchProcessor`.
5. THE App SHALL pass an `isProcessing` prop to `BatchProcessor` that is `true` while any batch job is running.

### Requirement 19: Batch Processing — Job Status Updates

**User Story:** As a content creator, I want to see each job's status update in real time as the batch runs, so that I know which video is being generated and which have completed.

#### Acceptance Criteria

1. WHEN `batchGenerate` begins processing a job, THE Store SHALL update that job's status to `'running'`.
2. WHEN a job completes successfully, THE Store SHALL update that job's status to `'complete'`.
3. WHEN a job fails with an error, THE Store SHALL update that job's status to `'error'` and store the error message in the job's `error` field.
4. THE `BatchProcessor` component SHALL re-render to reflect status changes as each job transitions through `pending` → `running` → `complete` / `error`.
5. THE Store SHALL expose the current batch job list as a reactive state value so the `BatchProcessor` component can display live status.

### Requirement 20: Batch Processing — Per-Job Output Storage

**User Story:** As a content creator, I want each completed batch job's video to be downloadable individually, so that I can retrieve any video from the batch after it finishes.

#### Acceptance Criteria

1. WHEN a batch job completes, THE Store SHALL store the completed `VideoProject` (including the `thumbnail` blob URL) in a `completedJobs` array in state.
2. THE `BatchProcessor` component SHALL display a "Download" button next to each completed job.
3. WHEN the user clicks "Download" for a completed job, THE BatchProcessor SHALL trigger a browser download of the job's video blob using the job's topic as the filename.
4. THE Store SHALL retain completed job data until the user explicitly clears the batch queue or resets the project.
5. IF a job's blob URL has been revoked (e.g. after a page reload), THEN THE BatchProcessor SHALL display the job as `'complete'` but disable the "Download" button with a tooltip "Video no longer available".

### Requirement 21: Batch Processing — Batch Progress Display

**User Story:** As a content creator, I want to see overall batch progress at a glance, so that I know how many videos have been completed out of the total.

#### Acceptance Criteria

1. WHILE `batchGenerate` is running, THE `BatchProcessor` component SHALL display a summary line in the format `"{completed}/{total} videos complete"`.
2. THE `BatchProcessor` component SHALL update the summary line after each job transitions to `'complete'` or `'error'`.
3. WHEN all jobs have finished (all are `'complete'` or `'error'`), THE `BatchProcessor` component SHALL replace the summary line with `"Batch complete — {complete_count} succeeded, {error_count} failed"`.
4. THE `BatchProcessor` component SHALL display the overall batch progress as a single progress bar that fills proportionally as jobs complete (e.g. 2 of 5 complete = 40% filled).

### Requirement 22: No New External APIs

**User Story:** As a developer, I want all improvements to work within the existing codebase dependencies, so that no new API keys, accounts, or third-party services are required.

#### Acceptance Criteria

1. THE system SHALL NOT introduce any new `fetch` calls to external domains not already present in the codebase.
2. THE system SHALL NOT add any new npm packages that communicate with external services.
3. THE system SHALL NOT require any new API keys or environment variables beyond those already defined in `.env.example`.
4. WHERE a feature requires data (e.g. description content, tags, thumbnail images), THE system SHALL derive that data exclusively from the `VideoProject` object and existing service functions.

### Requirement 23: Backward Compatibility

**User Story:** As a developer, I want all existing tests to continue passing after these changes, so that the improvements do not introduce regressions.

#### Acceptance Criteria

1. THE system SHALL preserve all existing public function signatures in `videoRenderer.ts`, `youtube.ts`, `thumbnail.ts`, `seoTitles.ts`, and `batchProcessor.ts` — new parameters SHALL be optional with backward-compatible defaults.
2. WHEN `generateYouTubeMetadata` is called with the existing three-parameter signature (`title`, `topic`, `script`), THE YouTubeMetadata generator SHALL produce valid output without throwing.
3. WHEN `generateSplitScreenThumbnail` is called, THE SplitScreenThumbnail generator SHALL always return a `Blob` (never throw to the caller) — errors SHALL be caught internally and the fallback `generateThumbnail` SHALL be called instead.
4. THE Renderer SHALL produce a valid video `Blob` for all existing quality presets (`draft`, `standard`, `high`) after the render speed changes are applied.
5. THE `BatchProcessor` class in `src/services/batchProcessor.ts` SHALL retain its existing `process`, `abort`, `getJobs`, `getCompletedJobs`, and `exportCompleted` methods with unchanged signatures.
