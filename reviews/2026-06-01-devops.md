## P0 â€” Critical / Blocks Deploy

### P0-1. No root `Dockerfile` â€” Docker-based deploys are completely broken

**Files:** `docker-compose.yml` (line 3), `.github/workflows/deploy.yml` (line 40)

Both `docker-compose.yml` (`build: .`) and the GitHub Actions deploy workflow (`context: .`) reference a Dockerfile at the repository root, but **no `Dockerfile` exists**. Only `tts-service/Dockerfile` is present.

- `docker-compose up` will fail immediately with "failed to solve: failed to read dockerfile: open /Dockerfile: no such file or directory".
- The CI "Build & Push" workflow will fail at the `docker/build-push-action` step.
- The only working deploy path is `scripts/deploy.sh` â†’ Railway CLI.

**Fix:** Create a multi-stage root `Dockerfile` (node:22-slim base, install native deps for canvas, build frontend, run as non-root). Or remove the Docker-based workflows if Railway is the sole target.

### P0-2. docker-compose health check URL mismatch â€” service will never become healthy

**File:** `docker-compose.yml` (line 20)

```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:5173/health"]
```

The actual health endpoint is `/api/health` (confirmed in `server/index.ts` line 94 and `railway.toml` line 6). The path `/health` will return the SPA fallback HTML (200 from static server) or a 404 â€” `curl -f` treats 404 as failure, and a 200 HTML response won't match `-f` semantics either since it's not a valid health response.

**Impact:** The `autotube` service will never report as healthy. Since `autotube` depends on `tts` with `condition: service_healthy`, and `tts` itself has a correct `/health` endpoint, the `tts` service will start but `autotube` will perpetually show as "starting" / "unhealthy".

**Fix:** Change to `http://localhost:5173/api/health`.

### P0-3. `deploy.sh` does not sync `package.json` or `package-lock.json`

**File:** `scripts/deploy.sh`

The deploy script rsyncs `server/`, `server-render/`, and copies `server.mjs`, `server-render.mjs`, `railway.toml`, `nixpacks.toml`, and `monitoring.ts` â€” but **never copies `package.json` or `package-lock.json`** into `deploy/`. The `deploy/package.json` is a separately maintained, git-tracked file that has already drifted from root:

| Dependency       | Root `package.json` | `deploy/package.json` |
|------------------|---------------------|-----------------------|
| `canvas`         | `^3.2.3`            | `^3.1.0`              |
| `better-sqlite3` | *not present*       | `^11.0.0`             |

Adding a new dependency to root `package.json` will silently not appear in production. This is a ticking time bomb.

**Fix:** Add `cp "$ROOT/package.json" "$DEPLOY_DIR/"` and `cp "$ROOT/package-lock.json" "$DEPLOY_DIR/"` to `deploy.sh`, or unify to a single `package.json`.

## P1 â€” Important

### P1-1. `canvas` version mismatch between root and deploy

**Files:** `package.json` (line 21), `deploy/package.json` (line 16)

Root has `canvas: ^3.2.3`, deploy has `canvas: ^3.1.0`. These ranges can resolve to different versions (e.g., root gets 3.2.3, deploy gets 3.1.x). Canvas 3.2.x introduced API changes that could cause runtime errors in production that don't reproduce locally.

**Fix:** Pin to the same version in both files, or better, have `deploy.sh` copy the root `package.json`.

### P1-2. No `tts-service/railway.toml`

**File:** `tts-service/` (directory)

The TTS service has a `Dockerfile` and `app.py` but **no `railway.toml`**. The `.env.local` references `KOKORO_SERVER_URL=https://tts-service-production-e96c.up.railway.app`, indicating this service is deployed to Railway. Without a `railway.toml`:
- No health check path configured (the app has `/health` at line 44 of `app.py`)
- No restart policy (if the ONNX model fails to load, the container won't retry)
- No start command override
- Railway will use defaults, which may not suit a 340MB model-loading service

**Fix:** Create `tts-service/railway.toml` with `healthcheckPath = "/health"`, `restartPolicyType = "on_failure"`, and appropriate `startCommand`.

### P1-3. `deploy.yml` tags Docker images with `latest` â€” not reproducible

**File:** `.github/workflows/deploy.yml` (line 35)

```yaml
tags: |
  type=sha,prefix=
  type=raw,value=latest
```

The `latest` tag is mutable and non-reproducible. If the commented-out deploy step is ever enabled (line 61: `docker pull ... :latest`), it could pull a different image than what was tested. Production deployments should use only immutable SHA-based tags.

**Fix:** Remove `type=raw,value=latest`. Use only `type=sha` and optionally `type=ref,event=tag`.

### P1-4. `tts-service/app.py` uses deprecated `@app.on_event("startup")`

**File:** `tts-service/app.py` (line 23)

```python
@app.on_event("startup")
async def startup_event():
```

FastAPI deprecated `@app.on_event("startup")` in favor of the `lifespan` context manager (Starlette 0.21+). This generates deprecation warnings and will be removed in a future FastAPI release. With `requirements.txt` specifying `fastapi>=0.100.0`, a future upgrade could break startup.

**Fix:** Migrate to `@asynccontextmanager` lifespan pattern.

### P1-5. `tts-service/requirements.txt` uses open-ended version ranges

**File:** `tts-service/requirements.txt`

```
fastapi>=0.100.0
uvicorn>=0.22.0
onnxruntime>=1.15.0
kokoro-onnx>=0.3.0
soundfile>=0.12.1
numpy<2.0.0
```

All packages use `>=` with no upper bound (except `numpy<2.0.0`). A breaking release of any dependency will be picked up on the next `pip install`, potentially breaking the build without any code change. This is especially risky for `onnxruntime` and `kokoro-onnx` which deal with model serialization.

**Fix:** Pin to specific versions or use compatible-release clauses (e.g., `fastapi~=0.100.0`).

### P1-6. `scripts/perf-budget.mjs` has broken import â€” will crash on execution

**File:** `scripts/perf-budget.mjs` (line 1)

```js
import { createHash, gzip } from 'node:crypto';
```

`gzip` is **not** an export of `node:crypto` â€” it belongs to `node:zlib`. In Node.js ESM, importing a non-existent named export throws `SyntaxError: The requested module 'node:crypto' does not provide an export named 'gzip'`. The script will crash immediately on load. Additionally, `createHash` is imported but never used (the code uses `createGzip` from line 2).

**Fix:** Remove line 1 entirely. The working import is already on line 2: `import { createGzip } from 'node:zlib'`.

### P1-8. 11 `__pycache__/*.pyc` files committed to git

**Files:** `deploy/server-render/vibevoice/vvembed/**/__pycache__/*.pyc`

11 Python bytecode files (`.cpython-314.pyc`) are committed to git in `deploy/server-render/vibevoice/`. These are:
- Platform/Python-version-specific (cpython-314 = Python 3.14)
- Regenerated automatically on import
- Bloat the repo with binary artifacts

The root `.gitignore` has no `__pycache__/` or `*.pyc` rule. The `deploy.sh` rsync excludes `__pycache__/` for new syncs, but already-committed files remain.

**Fix:** Add `__pycache__/` and `*.pyc` to `.gitignore`. Run `git rm -r --cached deploy/server-render/vibevoice/vvembed/**/__pycache__/`.

### P1-9. `better-sqlite3` in deploy but not in root â€” dependency asymmetry

**Files:** `deploy/package.json` (line 17), `package.json` (absent)

`deploy/package.json` includes `better-sqlite3: ^11.0.0` which is used by `server/migrate.mjs` (line 43: `await import("better-sqlite3")`). However, root `package.json` does not list this dependency. This means:
- Local development: `import("better-sqlite3")` will fail unless manually installed
- Deploy: works because `deploy/package.json` includes it
- `npm ci` in CI (which uses root `package-lock.json`) will not install `better-sqlite3`

**Fix:** Add `better-sqlite3` to root `package.json` dependencies, or make the migration script truly optional with a clear error message.

## P2 â€” Minor / Improvement

### P2-1. TTS Dockerfile runs as root â€” no `USER` directive

**File:** `tts-service/Dockerfile`

The container runs all processes as root. If the FastAPI app is compromised, the attacker has root access inside the container. Best practice is to create a non-root user.

**Fix:** Add `RUN useradd -m appuser && USER appuser` before the `CMD` directive.

---

### P2-2. TTS Dockerfile downloads model files without checksum verification

**File:** `tts-service/Dockerfile` (lines 17-20)

```dockerfile
RUN curl -L -o kokoro-v0_19.onnx https://github.com/thewh1teagle/kokoro-onnx/releases/download/model-files/kokoro-v0_19.onnx
```

340MB+ of model weights are downloaded over HTTPS without SHA256 verification. A compromised GitHub release or MITM attack could inject malicious model weights.

**Fix:** Add `sha256sum` verification after download.

---

### P2-3. `nixpacks.toml` uses `--break-system-packages` for pip

**File:** `nixpacks.toml` (line 24)

```toml
cmds = [
  "pip install --break-system-packages edge-tts",
  "npm run build"
]
```

The `--break-system-packages` flag bypasses PEP 668 protections. While functional, it could cause conflicts with system Python packages. A virtualenv would be cleaner and more portable.

**Fix:** Use `python3 -m venv /opt/venv && /opt/venv/bin/pip install edge-tts` and add `/opt/venv/bin` to PATH.

---

### P2-4. CI workflow does not install native deps for `canvas`

**File:** `.github/workflows/ci.yml`

The CI workflow runs `npm ci` on `ubuntu-latest` which will attempt to install `canvas` (v3.2.3). While `canvas` v3.x ships prebuilt binaries for common platforms, if the prebuilt is unavailable for the runner's Node/OS combination, it falls back to native compilation requiring `libcairo2-dev`, `libpango1.0-dev`, `libjpeg-dev`, `libgif-dev`, `librsvg2-dev`, and `pixman` dev headers â€” none of which are installed in the workflow.

**Fix:** Add an `apt-get install` step for canvas native deps, or add a fallback in case `canvas` install fails.

---

### P2-5. No Railway container size / memory configuration documented

**Files:** `railway.toml`, `nixpacks.toml`

The `railway.toml` has no `[deploy.resources]` section. The `docker-compose.yml` sets memory limits (2G for autotube, 1G for TTS), but Railway has no equivalent constraint. Given that:
- `canvas` + `ffmpeg` + `chromium` can easily consume 2-4GB during renders
- The Kokoro ONNX model alone is 340MB in memory
- Railway's default container may be too small

Without explicit resource configuration, the app may OOM-kill on Railway's default plan.

**Fix:** Add resource limits to `railway.toml` or document the required Railway plan.

---

### P2-6. `.env.example` references `VITE_KOKORO_SERVER_URL` but `.env.local` uses `KOKORO_SERVER_URL`

**Files:** `.env.example` (line 27), `.env.local` (line 5)

`.env.example` documents `KOKORO_SERVER_URL=` (no `VITE_` prefix), while `.env.local` uses `VITE_KOKORO_SERVER_URL=`. The `docker-compose.yml` passes `KOKORO_SERVER_URL` (no prefix). This inconsistency means developers may set the wrong variable name. Server-side code reads `KOKORO_SERVER_URL` (no prefix), while Vite-exposed client code would need `VITE_` prefix.

**Fix:** Clarify which prefix is correct for which context. Document both in `.env.example` if both are needed.
