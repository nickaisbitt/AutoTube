# Frontend/UX Review — 2026-06-01

## Summary
The frontend is well-structured with a clean pipeline-step architecture, good use of Zustand-pattern slices, and consistent industrial-design styling. However, there are critical issues: a broken VideoPlayer component with 11 suppressed type errors, a destructive "New Video" action lacking confirmation, invisible toast notifications, and a stale-closure bug in the playback keyboard handler. A significant amount of dead code (15+ unused components, 3 unused hooks) adds maintenance burden.

- Total issues: 18 (P0: 4, P1: 8, P2: 6)

---

## P0 — Must fix

### F-P0-1: VideoPlayer accepts 11 props it never uses, suppressed with @ts-ignore
- **File(s):** `src/components/PreviewStep/VideoPlayer.tsx:28-53`
- **Problem:** The component destructures 13 props but 11 of them are annotated with `// @ts-ignore - unused variable`. Props like `isPlaying`, `currentTime`, `onPlayPause`, `onMuteToggle`, `formatTime`, etc. are accepted from the parent but never referenced in the JSX. The video element uses native `controls` instead, making the entire custom playback UI dead code.
- **Why P0:** This means the storyboard-mode playback controls (play/pause, mute, time display, segment sync) are completely non-functional. Users in "MP4" mode see native browser controls instead of the intended custom UI. The type system is being actively lied to via 11 `@ts-ignore` directives.
- **Proposed fix:** Either implement the custom video controls using the passed props (removing `controls` from the `<video>` element), or remove the unused props from the interface and parent call site to match reality.
- **PR title:** `fix(frontend): remove @ts-ignore suppressions in VideoPlayer and wire up custom controls`
- **Diff size:** M
- **Cross-references:** Backend agent (video playback format), Testing agent (playback tests may be testing dead code)

### F-P0-2: "New Video" button destroys project without confirmation dialog
- **File(s):** `src/components/PreviewStep/index.tsx:182`, `src/store/index.ts:496-532`
- **Problem:** The "New Video" button (`data-testid="new-video-button"`) calls `onReset` which maps directly to `resetProject()`. This immediately revokes all blob URLs, clears the project, resets all step statuses, and wipes topic config — with zero confirmation. A user who accidentally clicks this loses their entire project.
- **Why P0:** Destructive, irreversible data loss from a single misclick. The `ConfirmDialog` component and `useConfirmDialog` hook already exist in the codebase (`src/components/ConfirmDialog.tsx`) but are never used anywhere.
- **Proposed fix:** Wire up `useConfirmDialog` (or inline a confirmation) before calling `resetProject()`. The infrastructure already exists.
- **PR title:** `fix(frontend): add confirmation dialog before resetting project`
- **Diff size:** S
- **Cross-references:** UX agent

### F-P0-3: Stale closure in usePlayback keyboard handler
- **File(s):** `src/components/PreviewStep/usePlayback.ts:193-223`
- **Problem:** The `useEffect` that registers keyboard shortcuts (Space, ArrowLeft, ArrowRight, m, r) has an empty dependency array `[]`, but it references `handlePlayPause` and `jumpToTime` which are plain functions recreated every render. The captured versions will always reference the initial render's `isPlaying`, `currentTime`, and `currentSegmentIndex` state values. Pressing Space will always think `isPlaying` is `false` (its initial value).
- **Why P0:** Keyboard shortcuts for play/pause and seeking are broken in the preview step. Users pressing Space will get inconsistent behavior.
- **Proposed fix:** Either wrap `handlePlayPause` and `jumpToTime` in `useCallback` and add them to the dependency array, or use a ref pattern (like `handlersRef` in `useKeyboardShortcuts.ts`) to always call the latest version.
- **PR title:** `fix(frontend): fix stale closure in preview keyboard shortcut handler`
- **Diff size:** S
- **Cross-references:** None

### F-P0-4: ToastContainer is never mounted — all toast notifications are invisible
- **File(s):** `src/components/Toast.tsx:133-145`, `src/main.tsx`, `src/App.tsx`
- **Problem:** `ToastContainer` (the default export of `Toast.tsx`) is never rendered anywhere in the component tree. The `addToast()` function pushes toasts into a module-level queue and notifies listeners, but `useToastState()` is only called inside `ToastContainer`, which is never mounted. The `useCopyToClipboard` hook and various other places call `toast()` but users never see any notification.
- **Why P0:** All user-facing toast feedback (copy confirmations, error messages, success notifications) is silently dropped. Users get no feedback for actions like clipboard copy, export success, etc.
- **Proposed fix:** Add `<ToastContainer />` to `App.tsx` or `AppShell.tsx`.
- **PR title:** `fix(frontend): mount ToastContainer in app tree so notifications are visible`
- **Diff size:** S
- **Cross-references:** None

## P1 — Should fix soon

### F-P1-1: Unsafe `as unknown as` type cast on assembleVideo in two locations
- **File(s):** `src/App.tsx:28-31`, `src/components/PipelineStepRouter.tsx:94-97`
- **Problem:** Both `App.tsx` and `PipelineStepRouter.tsx` cast `assembleVideo` through `as unknown as` to change its signature. The actual store function accepts `(exportOptions?, projectOverride?)` but the cast pretends it has a different type. This defeats TypeScript's safety and means the actual function signature in the store doesn't match what callers expect.
- **Why P1:** Type safety is bypassed at two critical call sites. If the store function signature changes, neither call site will get a compile error.
- **Proposed fix:** Fix the `assembleVideo` type in `store/index.ts` to match its actual usage, removing the need for casts.
- **PR title:** `fix(frontend): align assembleVideo type signature to eliminate unsafe casts`
- **Diff size:** S
- **Cross-references:** Architecture agent

### F-P1-2: ExportModal and OnboardingModal lack focus traps
- **File(s):** `src/components/ExportModal.tsx:44-196`, `src/components/OnboardingModal.tsx:51-129`
- **Problem:** `SettingsModal` correctly implements a focus trap (lines 39-67) with Tab/Shift+Tab cycling and focus restoration. `ExportModal` and `OnboardingModal` do not. Users can Tab out of these modals into background content, and focus is not restored on close.
- **Why P1:** Accessibility regression — keyboard users and screen reader users can escape the modal context. WCAG 2.1.2 violation.
- **Proposed fix:** Extract the focus trap logic from `SettingsModal` into a shared `useFocusTrap` hook and apply it to all three modals.
- **PR title:** `fix(frontend): add focus traps to ExportModal and OnboardingModal`
- **Diff size:** M
- **Cross-references:** Accessibility agent

### F-P1-6: 74+ uses of `transition-colors duration-200` are globally nullified by index.css
- **File(s):** `src/index.css:51-54`, and 74+ component locations

- **Proposed fix:** Either remove the global transition kill-switch (if transitions are desired), or strip the `transition-*` classes from all components (if the instant feel is intentional). Document the design decision.
- **PR title:** `style(frontend): reconcile transition classes with global CSS kill-switch`

## Observations

### Things that look good
- **Slice architecture** — The store is well-decomposed into `projectSlice`, `pipelineSlice`, `configSlice`, `narrationSlice`, and `uiSlice`, each with clear interfaces. The composed hook maintains backward compatibility.
- **Abort controller pattern** — Every pipeline step has its own `AbortController` ref, enabling clean cancellation. The `cancelCurrentOperation` function correctly routes to the active step's controller.
- **Stuck-state watchdog** — The 300-second watchdog in `pipelineSlice.ts:56-79` that auto-cancels stuck renders is a great defensive pattern.
- **Error boundaries** — Both a global `ErrorBoundary` and per-step `StepErrorBoundary` wrappers exist, preventing full-app crashes.
- **Secure key management** — The `configSlice` implements PIN-encrypted API key storage with session/encrypted blob separation. Good security hygiene for a browser app.
- **Project validation** — `validateStoredProject()` in `store/index.ts` thoroughly validates localStorage data, handles migrations, and discards stale projects (>24h).
- **Consistent design system** — The industrial/brutalist theme is applied consistently via `index.css` overrides and a well-defined color palette in `@theme`.
- **Good aria coverage** — 74+ aria-label/aria-modal/role attributes across components. Most interactive elements have proper labels.

### Tech debt not worth a PR
- The `narrationSlice` is misnamed — it only holds `sourcingRef` which is used for media sourcing, not narration. Renaming would touch many files for minimal benefit.
- The `StepErrorBoundary` class component in `PipelineStepRouter.tsx` could be extracted to its own file, but it's only 10 lines.
- `BatchProcessor.tsx` has two different job types (`LocalBatchJob` and `StoreBatchJob`) unified via a union type with a `'project' in job` discriminator — functional but inelegant.
- The `quality` parameter type mismatch between `ExportModal` (uses `keyof typeof QUALITY_PRESETS`) and `App.tsx` (uses `'draft' | 'standard' | 'high'`) works because they resolve to the same union, but it's fragile.
